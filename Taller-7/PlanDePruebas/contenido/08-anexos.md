## Glosario

| Término | Definición |
|---|---|
| **Bug / Defecto** | Diferencia entre el comportamiento observado y el esperado según especificación |
| **Caso de Prueba (TC)** | Conjunto de precondiciones, pasos, datos de prueba y resultado esperado para verificar un comportamiento |
| **Criterio de Aceptación** | Condición verificable que debe cumplirse para dar por completada una HU o un plan de pruebas |
| **HU** | Historia de Usuario: descripción funcional desde la perspectiva del usuario |
| **Milestone** | Agrupador de issues en GitHub que representa un Test Run o entrega |
| **Prioridad** | Urgencia de ejecución o corrección: alta, media, baja |
| **Prueba de caja negra** | Técnica que evalúa comportamiento externo sin conocer implementación interna |
| **Prueba funcional** | Verifica que el sistema cumple los requisitos funcionales especificados |
| **Prueba de regresión** | Verifica que cambios no rompen funcionalidades existentes |
| **Prueba de humo (smoke)** | Conjunto mínimo de pruebas para validar que el sistema arranca y sus flujos críticos funcionan |
| **Severidad** | Impacto del defecto sobre el sistema: critical, major, minor, trivial |
| **Test Run** | Ejecución planificada de un conjunto de casos de prueba en un ambiente específico |
| **Trazabilidad** | Capacidad de enlazar un caso de prueba con su HU, CU o requisito de origen |
| **RACI** | Matriz de responsabilidades: Responsable, Aprobador, Consultado, Informado |
| **PENDIENTE** | Estado de una solicitud de vendedor recién registrada, aún no revisada |
| **CIFIN / Datacredito** | Centrales de riesgo crediticio consultadas durante la revisión de solicitudes de vendedor |
